# 会议助手

## 真实工作流

会议助手真正有价值的地方，不是把几句话扩写成纪要，而是把会议前后的信息闭环做完整。

| 维度 | 真实情况 |
|------|----------|
| 触发点 | 上线协调会、需求评审、项目复盘、客户会议等需要会前准备和会后追踪的会议 |
| 现有材料 | 会议目标、参会角色、议题、手写笔记、录音转写、会中决议 |
| 卡点 | 会前议题没排好，会后决议和待办混在一起，负责人和截止时间容易遗漏 |
| DesireCore 介入 | 会前生成议程和待决策项，会后整理纪要、待办表和跟进清单 |
| 验收结果 | 主持人得到一份能发给参会人的纪要，以及可同步到项目管理工具的任务清单 |

## 完整对话示例

```
你: 明天下午有个产品上线协调会，参会人有产品经理、开发负责人、
    测试经理和运维经理。主要讨论上线时间和灰度方案。
    帮我准备一份议程。

AI 助手: 好的，以下是会议议程草案：

        产品上线协调会
        时间：明天下午 14:00-15:30
        地点：[待确认]

        议程安排：
        14:00-14:10  开场：项目进展概述（产品经理）
        14:10-14:30  开发完成情况及遗留问题（开发负责人）
        14:30-14:50  测试进展与质量报告（测试经理）
        14:50-15:10  上线方案与灰度策略（运维经理）
        15:10-15:25  讨论：确认上线时间和回滚方案
        15:25-15:30  总结：明确各方待办和截止时间

        待决策事项：
        1. 上线时间选择（周五下午 or 下周一上午）
        2. 灰度比例（10% → 50% → 100%）
        3. 回滚触发条件

        需要调整吗？

你: 议程没问题。会开完了，我把记的笔记发给你，帮我整理纪要。
    "决定下周一上线，先灰度 10%，观察 2 小时没问题再扩到 50%，
    隔天全量。开发这边还有 2 个 P1 的 bug 要周五前修完，
    测试周末加班做回归。运维准备回滚脚本，周日前完成。"

AI 助手: 会议纪要已整理完毕：

        会议纪要
        会议主题：产品上线协调会
        日期：[今天日期]

        一、决议事项
        1. 上线时间：下周一上午
        2. 灰度策略：10% → 观察 2 小时 → 50% → 隔天 100%
        3. 回滚方案：运维准备自动回滚脚本

        二、待办事项
        | 事项 | 负责人 | 截止时间 |
        |------|--------|---------|
        | 修复 2 个 P1 Bug | 开发负责人 | 周五 |
        | 回归测试 | 测试经理 | 周末 |
        | 准备回滚脚本 | 运维经理 | 周日 |

        三、下次跟进
        周一上午 9:00 上线前最终确认

        需要我整理成可发送版本吗？
```

## 关键步骤

1. **会前对齐目标** -- 写清会议要产出什么：决策、方案评审、风险同步还是任务分配
2. **带着上次待办开会** -- 把上次行动项、未关闭事项和新议题一起放进议程
3. **会中区分事实和决议** -- 笔记里区分讨论内容、已确认决议、待确认问题和风险
4. **行动项结构化** -- 每个待办都要有任务、唯一负责人、截止时间、完成标准和依赖项
5. **会后进入跟进队列** -- 纪要不只是发送，还要把行动项同步到项目工具或下次会议议程

## 最终成果

一份结构清楚的会议纪要草稿，包含决议事项、待办清单、负责人、截止时间、完成标准和下次跟进安排，可以继续改成邮件版、飞书公告版或项目任务清单。

:::tip 最佳实践
- 会议笔记不需要完整记录每句话，但关键决定、负责人、截止时间要尽量写清楚
- 如果有会议录音转写文稿，直接粘贴给智能体效果更好
- 纪要格式可以让智能体记住你的偏好（比如"每次都用表格列出待办"），下次就不用重复说明了
- 会议行动项如果没有唯一负责人和完成标准，后续很容易变成“大家都知道但没人推进”的事项
:::
